# 安全风险分析与解决方案

## 一、当前安全风险分析

### 1.1 管理入口暴露 (严重)

**问题位置**: [Layout.tsx](file:///d:/File/workSpace/AI-test/lvjiang-cup-test/frontend/src/components/layout/Layout.tsx#L184-L197)

主页面顶部导航栏右上角有明确的"管理"链接,直接暴露了管理后台入口 `/admin/login`。

```tsx
<Link to="/admin/login" className="...">
  <span>管理</span>
</Link>
```

**风险**: 任何用户都可以直接访问管理登录页面,增加了暴力破解和针对性攻击的风险。

### 1.2 管理页面与游客页面未分离 (高危)

**问题描述**: 前端是一个单页应用 (SPA),所有页面组件(包括管理后台)都打包在同一个 JavaScript bundle 中。

**受影响文件**:
- [App.tsx](file:///d:/File/workSpace/AI-test/lvjiang-cup-test/frontend/src/App.tsx) - 所有路由集中定义
- [router/index.tsx](file:///d:/File/workSpace/AI-test/lvjiang-cup-test/frontend/src/router/index.tsx) - 路由配置
- 所有 `src/pages/admin/` 下的管理页面组件

**风险**:
- 管理员页面组件源码(如 [Dashboard.tsx](file:///d:/File/workSpace/AI-test/lvjiang-cup-test/frontend/src/pages/admin/Dashboard.tsx))会下发给所有用户
- 攻击者可以通过分析前端代码,发现所有 API 调用路径和参数结构
- 虽然前端有 `ProtectedRoute` 保护,但仅依赖 Token 验证,源码本身是公开的

### 1.3 Swagger API 文档暴露 (高危)

**问题位置**: [main.ts](file:///d:/File/workSpace/AI-test/lvjiang-cup-test/backend/src/main.ts#L130-L137)

```typescript
const document = SwaggerModule.createDocument(app, config);
SwaggerModule.setup('api/docs', app, document);
```

**风险**: 生产环境中任何人都可以访问 `/api/docs`,查看完整的 API 结构,包括:
- 所有管理员操作端点 (`POST /api/admin/reset-slots`, `DELETE /api/admin/data`)
- 请求/响应格式
- 认证机制

### 1.4 管理 API 路径可预测 (中危)

**问题描述**: 所有管理端点遵循明显的路径规律:
- `/api/admin/teams` - 战队管理
- `/api/admin/matches` - 比赛管理
- `/api/admin/init-slots` - 初始化槽位
- `/api/admin/reset-slots` - 重置槽位
- `/api/admin/data` - 清空所有数据

**风险**: 即使不知道前端代码,攻击者也可以通过猜测路径发现管理 API。

### 1.5 缺少 IP 访问限制 (中危)

**问题描述**: 管理端点没有任何 IP 白名单限制,任何人都可以尝试访问。

## 二、解决方案

### 方案 A: 快速修复方案 (推荐优先实施, 2-3 天)

适用于快速降低风险,不需要大规模重构。

#### A1. 隐藏管理入口

**修改文件**: [Layout.tsx](file:///d:/File/workSpace/AI-test/lvjiang-cup-test/frontend/src/components/layout/Layout.tsx)

移除公开的"管理"链接,改为通过特殊路径访问:

```tsx
// 方案 1: 完全移除管理入口链接
// 方案 2: 保留链接但改为隐藏路径,如输入特定快捷键才显示
```

**实施方式**:
- 从顶部导航栏移除"管理"按钮
- 管理员通过直接访问隐藏路径进入,如: `/system/console/login`
- 或使用键盘快捷键 (如 `Ctrl+Shift+A`) 显示管理入口

#### A2. 禁用生产环境 Swagger

**修改文件**: [main.ts](file:///d:/File/workSpace/AI-test/lvjiang-cup-test/backend/src/main.ts)

```typescript
// 仅在开发环境启用 Swagger

  const config = new DocumentBuilder()
    .setTitle('驴酱杯赛事 API')
    .setDescription('驴酱杯LOL娱乐赛事网站后端 API 文档')

    .addBearerAuth()
    .build();

```

#### A3. 添加环境变量控制

**新增环境变量** (`.env.production`):

```env
# 管理后台隐藏路径前缀 (默认: admin)
ADMIN_ROUTE_PREFIX=system-console-8f3a

# 是否启用 Swagger (生产环境应为 false)
ENABLE_SWAGGER=false

# 管理端点 IP 白名单 (逗号分隔,留空表示不限制)
ADMIN_IP_WHITELIST=

# 管理端点访问密钥 (用于额外的请求头验证)
ADMIN_ACCESS_KEY=your-secret-key-here
```

#### A4. 添加 IP 白名单守卫

**新建文件**: `backend/src/common/guards/ip-whitelist.guard.ts`

```typescript
import { Injectable, CanActivate, ExecutionContext, ForbiddenException } from '@nestjs/common';

@Injectable()
export class IpWhitelistGuard implements CanActivate {
  canActivate(context: ExecutionContext): boolean {
    const whitelist = process.env.ADMIN_IP_WHITELIST;
    if (!whitelist) return true; // 未配置白名单时不限制

    const request = context.switchToHttp().getRequest();
    const ip = request.ip || request.connection.remoteAddress;
    
    const allowedIPs = whitelist.split(',').map(ip => ip.trim());
    if (!allowedIPs.includes(ip)) {
      throw new ForbiddenException('IP 地址未授权');
    }
    return true;
  }
}
```

**修改文件**: [App.tsx](file:///d:/File/workSpace/AI-test/lvjiang-cup-test/frontend/src/App.tsx), [router/index.tsx](file:///d:/File/workSpace/AI-test/lvjiang-cup-test/frontend/src/router/index.tsx)

### 方案 B: 前后端分离方案 (推荐长期方案, 5-7 天)

将管理后台与游客页面完全分离为两个独立应用。

#### B1. 前端应用拆分

```
frontend/                    # 游客只读页面 (公开访问)
├── src/
│   ├── pages/
│   │   └── Home.tsx
│   ├── components/
│   │   └── features/        # 游客可见组件
│   └── App.tsx

frontend-admin/              # 管理后台 (独立应用, 需认证)
├── src/
│   ├── pages/
│   │   └── Dashboard.tsx
│   │   └── Teams.tsx
│   │   └── ...
│   ├── components/
│   │   └── admin/           # 管理员专用组件
│   └── App.tsx
```

**优势**:
- 管理后台代码不会下发给普通用户
- 可以部署在不同域名,如:
  - 游客页面: `https://example.com`
  - 管理后台: `https://admin.example.com` 或 `https://internal.example.com`
- 管理后台可以添加额外的网络层保护 (如 VPN, IP 白名单)

#### B2. 后端 API 分层

```
公开 API (无需认证):
  GET  /api/teams
  GET  /api/matches
  GET  /api/streams
  GET  /api/videos
  GET  /api/matches/:id/games/:num

管理 API (需要认证 + 可选IP白名单):
  POST   /api/internal/auth/login
  PUT    /api/internal/teams/:id
  DELETE /api/internal/teams/:id
  POST   /api/internal/admin/init-slots
  ...
```

**修改点**:
- 在 NestJS 中使用全局前缀区分公开和管理端点
- 管理端点添加 `@UseGuards(JwtAuthGuard, IpWhitelistGuard)`
- 可选: 添加 API 版本控制 (`/api/v1/internal/...`)

#### B3. Docker 部署调整

**修改文件**: [docker-compose.yml](file:///d:/File/workSpace/AI-test/lvjiang-cup-test/deploy/docker-compose.yml)

```yaml
services:
  # 前端 (游客页面)
  frontend:
    image: .../frontend:${TAG}
    ports:
      - "80:3001"

  # 管理后台 (限制访问)
  frontend-admin:
    image: .../frontend-admin:${TAG}
    ports:
      - "127.0.0.1:8080:3002"  # 仅本地可访问
    environment:
      - ADMIN_IP_WHITELIST=${ADMIN_IP_WHITELIST}

  # 后端
  backend:
    image: .../backend:${TAG}
    # 管理端点通过 IP 白名单保护
```

## 四、注意事项

1. **环境变量安全**: 不要将 `.env.production` 提交到 Git, 通过 CI/CD 注入
2. **HTTPS**: 确保生产环境使用 HTTPS, 防止 Token 被中间人窃取
3. **Token 安全**: 考虑使用 httpOnly cookie 替代 localStorage 存储 Token
4. **定期更新**: JWT Secret 应定期更换, 建议最长 90 天
5. **备份**: 修改前务必备份数据库
